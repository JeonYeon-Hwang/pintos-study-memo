file system을 읽고 하려곡 하지 말 것....  프로그램은 file에 저장되어 있으므로 이를 가져와서 실행되는게 보통임.
하드디스크: 물리적 개념 <=> file sys: OS가 해석하는 방식

예시: fat => ntfs ... os 마다 좀 다름.
현재는 간단한 형태의 file sys는 pintos에 구현이 되어 있는 상태 => 현재로서는 제약사항이 많다... 동기화 방법이 없음(요거만 읽기)

만약 달랐다면? linux 실행 환경 기준으로 컴파일 되고, pintos는 이를 읽지 못하는 문제 발생.

여기서 `args-single onearg`에서 파라미터를 받아서 실행하는 환경을 구성해야 하는 것이 이번 목표.

도식화 살펴보며 ... 
load()함수: 메모리에 올리는 함수 ... argument passing이 잘 못되고 있음.
... 규칙에 맞게 차곡차곡 쌓도록 구현할 것. 어려운 이유: 얼라인먼트? 
do_iret()함수: 커널이 아닌 유저 모드로 실행.

커널 모드 vs 유저 모드
응용 프로그램이 죽더라도 => 커널이 죽지 않도록 하기 위해
중요한 일은 커널에서 하고, 유저 프로그램에 던져주는 방식으로 분리.
서로 왔다 갔다 하는 명령어가 있다. 막 뜯어볼 필요는 없다.
C코드의 printf메서드를 보면, syscall이 있고, 이걸 이번 프로젝트에서 구현을 해야 한다.
프로젝트 1에서는 커널에서만 printf가 돌아갔지만, 이를 유저에서 커널로 printf롤 전달을 하도로 해야 함.
... 많은 부분이 해결 될 것.

**메모리 구조**
일반 상식처럼.. 메모리를 각 영역을 쪼개서 특정 부위를 사용하는 방식이 아님.
커널이 쓰는 ... 유저가 쓰는 메모리 영역을 침범하지 않도록 해야 함. validation check 수시로 해야 함.
유저가 쓰는 포인터는 항시 validation check하여 침범하지 못하도록.

fork() 구현할 때.. semaphore와 lock을 사용하여 동기화할 필요가 있음.
process_main/wait?이 아무것도 구현 안되어 있음.
sys_call의 handler 부문에도 구현 필요.
가장 어려운 테스트는? ... out of memory. 잘못 해석해서 그런 게 특징.
아까는 pass였다가 추가 구현하여 fail 되면? => 동기화 문제일 가능성이 높음.

**피드백 하기(포스트 모텀?)**
누군가에게 피드백 시 blame 하는 방식은 ... 되돌아오는 것이 없다.

우선: 잘하는 것 서로 칭찬?같은 거를 한 다음에..
이후: what's wrong => what's if?로 바꾸는 것.
전자는 지적질에 가깝다. 후자는, 뭘 개선을 하면 좋을까를 이야기 하는 것.
먼 이야기가 아닌, 다음 주에 action이 가능한 것으로 이야기 할 것.

회고를 하는 방법:
specific - 구체적일 것.
merserable - 측정이 가능할 것. 개선을 보여줄 수 있음.
accounable - 팀 내에서 달성이 가능한 것에 한정.
result, time - 트래킹이 가능하고 결과가 나오는 게 가능한 것으로.

포스트 잇 .. 같은 것을 준비

=> 종합하여, 실제 좋은 것? 산출 가능.
